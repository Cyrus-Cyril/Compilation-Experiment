## PERF-001 p07_loop / p08_basic_combined / p09_advanced_graph / p10_advanced_matrix 运行超时

- **时间**: 2026-07-04
- **标签**: `performance`, `optimization`, `backend`, `blocker`
- **关联文档**: `docs/任务要求.md`, `docs/design/architecture.md`

### 现象

4 个性能测试用例因运行超时（超过评测系统时间限制）未完成编译或执行，得分为 0.00：

| 用例 | 状态 | 执行时间 | 评分百分比 | 得分 |
|------|------|----------|------------|------|
| `p07_loop` | 运行超时 | - | - | 0.00 |
| `p08_basic_combined` | 运行超时 | - | - | 0.00 |
| `p09_advanced_graph` | 运行超时 | - | - | 0.00 |
| `p10_advanced_matrix` | 运行超时 | - | - | 0.00 |

### 影响

- **测试范围受阻**: 4 个评测用例完全无法得分，合计损失约 40% 性能测试分数
- **性能瓶颈未暴露**: 由于超时发生在编译或执行阶段，无法获取执行时间数据来判断进一步的优化空间
- **用户信心受损**: 基础循环和综合程序无法在合理时间内完成，说明编译器存在严重的性能缺陷

### 影响范围

- 所有包含循环/迭代的 ToyC 程序（`p07_loop`、`p08_basic_combined`）
- 图算法相关程序（`p09_advanced_graph`）
- 矩阵运算相关程序（`p10_advanced_matrix`）
- 间接影响：可能暴露 IR 生成或后端代码生成中存在的指数级复杂度问题

### 复现环境

- 评测系统环境：RISC-V 交叉编译，Spike 模拟器
- 优化选项：`-opt`（启用优化 Pass）

### 复现步骤

1. 在评测系统上使用 `-opt` 选项编译对应测试用例
2. 期望：在合理时间限制内完成编译并正确运行
3. 实际：编译器或生成的可执行文件运行超时，被评测系统终止

### 排查方向

1. **确认超时阶段**: 需要区分超时发生在编译器阶段还是生成的可执行文件运行阶段
   - 若为编译阶段超时：问题在优化 Pass（IR 优化器）存在算法复杂度问题
   - 若为运行阶段超时：问题在后端代码生成（生成的 RISC-V 汇编效率太低）或缺乏关键优化

2. **检查优化 Pass 实现**: 当前优化 Pass 实现可能存在以下问题：
   - 未实现循环不变式外提（Loop Invariant Code Motion）
   - 未实现循环展开（Loop Unrolling）
   - 未实现强度削弱（Strength Reduction）
   - 优化 Pass 本身存在 O(n^2) 或更高复杂度的算法

3. **检查后端代码生成**: 对于循环密集型程序，后端生成的汇编可能存在：
   - 过多的栈访问（频繁的 `lw`/`sw` 操作）
   - 不必要的寄存器溢出
   - 未利用 RISC-V 的特性（如乘加指令）

### 复现输入（推测）

根据测试名称推测各测试的内容：

**p07_loop** — 循环密集型基准测试：
```c
int main() {
    int sum = 0;
    int i = 0;
    while (i < 1000000) {
        sum = sum + i;
        i = i + 1;
    }
    return sum;
}
```

**p08_basic_combined** — 基础综合测试（循环 + 分支 + 函数调用组合）。

**p09_advanced_graph** — 图算法测试（涉及多重嵌套循环和数组/指针访问）。

**p10_advanced_matrix** — 矩阵运算测试（多重嵌套循环 + 大量局部变量）。

### 建议修复方向

1. **实现循环优化 Pass**（优先级最高）：
   - 循环不变式外提（LICM）：将循环内不变的计算移到循环外
   - 循环展开：减少循环控制开销
   - 强度削弱：将乘法/除法替换为移位/加法

2. **实现常量传播与折叠**：
   - 全局常量传播（Global Constant Propagation）
   - 常量折叠（Constant Folding）

3. **实现死代码消除**：
   - 消除无用的 IR 指令和基本块

4. **优化寄存器分配**：
   - 减少循环体内的栈访问次数
   - 优先将循环体内频繁使用的变量分配到寄存器

## PERF-002 p04_common_subexpr / p06_tail_recursion / p11_global_const_prop 优化比例偏低

- **时间**: 2026-07-04
- **标签**: `performance`, `optimization`, `backend`
- **关联文档**: `docs/任务要求.md`, `docs/design/ir_design.md`, `docs/design/architecture.md`

### 现象

以下 3 个已通过的测试用例虽然执行结果正确，但与参考实现的性能差距较大，优化比例（评分百分比）偏低：

| 用例 | 状态 | 执行时间 | 评分百分比 | 得分 |
|------|------|----------|------------|------|
| `p04_common_subexpr` | 通过 | 6124ms | 11% | 0.99 |
| `p06_tail_recursion` | 通过 | 7042ms | 9% | 0.79 |
| `p11_global_const_prop` | 通过 | 6828ms | 11% | 0.94 |

### 影响

- **得分偏低**: 这三个用例的总得分仅为 2.72（满分 6.00），损失约 55% 的应得分数
- **公共子表达式未消除**: `p04_common_subexpr` 的 11% 说明编译器完全没有进行公共子表达式消除（CSE）
- **尾调用未优化**: `p06_tail_recursion` 的 9% 说明编译器没有实现尾递归优化（TCO）
- **常量传播未生效**: `p11_global_const_prop` 的 11% 说明全局常量传播优化缺失或不完整

### 影响范围

- 所有包含重复计算的表达式（CSE 缺失）
- 所有尾递归函数（TCO 缺失）
- 所有使用了全局常量的程序（常量传播缺失）

### 复现步骤

1. 在评测系统上使用 `-opt` 选项编译对应测试用例
2. 观察执行时间与参考实现的对比
3. 优化比例显著偏低，说明编译器与参考实现存在指数级性能差距

### 排查方向

1. **确认当前优化 Pass 列表**: 确认当前 `-opt` 模式下实际运行了哪些优化 Pass
2. **检查 IR 优化器入口**: 确认 `Optimizer::run()` 是否调用了预期的优化 Pass
3. **检查每个优化的实现完整性**: 确认各优化是否真正生效

### 复现输入（推测）

**p04_common_subexpr** — 公共子表达式测试：
```c
int main() {
    int a = 1, b = 2, c = 3, d = 4;
    int x = (a + b) * c + (a + b) * d;  // a+b 出现两次，可 CSE
    int y = (c - d) * a + (c - d) * b;  // c-d 出现两次，可 CSE
    return x + y;
}
```

**p06_tail_recursion** — 尾递归测试：
```c
int sum(int n, int acc) {
    if (n == 0) return acc;
    return sum(n - 1, acc + n);  // 尾递归，可优化为循环
}
int main() {
    return sum(100000, 0);
}
```

**p11_global_const_prop** — 全局常量传播测试：
```c
const int N = 10000;
const int M = 20000;
int main() {
    int sum = 0;
    int i = 0;
    while (i < N) {      // N 可折叠为 10000
        sum = sum + M;   // M 可折叠为 20000
        i = i + 1;
    }
    return sum;
}
```

### 建议修复方向

1. **实现公共子表达式消除（CSE）**：
   - 在 IR 级别识别重复计算
   - 将重复的计算替换为对首次计算结果的引用
   - 注意处理基本块边界和寄存器生命周期

2. **实现尾递归优化（TCO）**：
   - 在 IR 级别检测尾递归调用（`CALL` 指令后紧接 `RET`）
   - 将尾递归替换为循环（更新参数并跳转到函数入口）
   - 需要处理多个尾递归出口的情况

3. **实现全局常量传播**：
   - 基于数据流分析的常量传播
   - 在 CFG 上执行前向数据流分析
   - 将 `LOAD_GLOBAL` 常量替换为立即数

4. **完善常量折叠**：
   - 确保算术指令（`ADD`, `SUB`, `MUL`, `DIV`, `MOD`）在两个操作数均为常量时折叠
   - 确保比较指令（`EQ`, `NE`, `LT`, `GT`, `LE`, `GE`）在操作数均为常量时折叠
   - 确保逻辑指令（`AND`, `OR`, `NOT`）在操作数均为常量时折叠

## PERF-003 p01_const / p02_dead_code / p03_copy / p05_algebra 未达到预期优化效果

- **时间**: 2026-07-04
- **标签**: `performance`, `optimization`, `backend`
- **关联文档**: `docs/任务要求.md`, `docs/design/ir_design.md`

### 现象

以下 4 个测试虽然通过，但优化比例普遍偏低（3%-6%），与预期优化效果差距显著：

| 用例 | 状态 | 执行时间 | 评分百分比 | 得分 |
|------|------|----------|------------|------|
| `p01_const` | 通过 | 5549ms | 6% | 0.51 |
| `p02_dead_code` | 通过 | 11658ms | 3% | 0.26 |
| `p03_copy` | 通过 | 8500ms | 4% | 0.40 |
| `p05_algebra` | 通过 | 13039ms | 3% | 0.32 |

### 影响

- **得分偏低**: 这四个用例的总得分仅为 1.49（满分 8.00），损失约 81% 的应得分数
- **常量折叠不完整**: `p01_const` 的 6% 说明常量折叠优化仅实现了极小部分
- **死代码未消除**: `p02_dead_code` 的 3% 说明死代码消除几乎没有效果
- **拷贝传播未实现**: `p03_copy` 的 4% 说明拷贝传播优化缺失
- **代数化简未实现**: `p05_algebra` 的 3% 说明代数化简优化缺失

### 影响范围

- 所有包含常量表达式的程序
- 所有包含死代码（无用赋值、无用变量）的程序
- 所有包含拷贝链（`a = b; c = a;`）的程序
- 所有包含代数恒等式的程序（`x + 0`, `x * 1`, `x - x` 等）

### 复现步骤

1. 在评测系统上使用 `-opt` 选项编译对应测试用例
2. 观察执行时间与参考实现的对比
3. 优化比例极低，说明编译器基本没有进行有效的优化

### 排查方向

1. **检查优化 Pass 的触发条件**: 确认每个优化是否被正确触发
2. **检查 IR 指令模式匹配**: 确认优化 Pass 是否能正确识别可优化的指令模式
3. **检查优化 Pass 的执行顺序**: 某些优化依赖于其他优化的结果
4. **检查常量折叠的完备性**: 确认所有算术、比较、逻辑指令的常量折叠都正确处理

### 复现输入（推测）

**p01_const** — 常量优化测试：
```c
int main() {
    int a = 10;
    int b = 20 + 30;       // 可折叠为 50
    int c = a * 2 + 100;   // 可折叠为 120
    int d = c / 2;         // 可折叠为 60
    return d - 10;         // 可折叠为 50
}
```

**p02_dead_code** — 死代码消除测试：
```c
int main() {
    int a = 10;            // 死赋值（a 未使用）
    int b = 20;            // 死赋值（b 被覆盖）
    b = 30;
    int c = a;             // a 是死代码
    int d = 40;
    return d;              // 只有 d 被使用
}
```

**p03_copy** — 拷贝传播测试：
```c
int main() {
    int a = 42;
    int b = a;             // 拷贝，可传播
    int c = b;             // 拷贝，可传播
    int d = c;             // 拷贝链，可折叠
    return d;              // 直接返回 42
}
```

**p05_algebra** — 代数化简测试：
```c
int main() {
    int a = 100;
    int b = a + 0;         // 可化简为 a
    int c = a * 1;         // 可化简为 a
    int d = a - a;         // 可化简为 0
    int e = a * 0;         // 可化简为 0
    int f = a / 1;         // 可化简为 a
    return b + c + d + e + f;
}
```

### 建议修复方向

1. **完善常量折叠（Constant Folding）**：
   - 在 `visitBinaryExpr` 或 IR 优化 Pass 中检测两个操作数均为常量的算术指令
   - 覆盖所有指令类型：`ADD`, `SUB`, `MUL`, `DIV`, `MOD`, `EQ`, `NE`, `LT`, `GT`, `LE`, `GE`, `AND`, `OR`, `NOT`
   - 注意处理除数为零的情况

2. **实现死代码消除（Dead Code Elimination）**：
   - 基于活跃变量分析（Live Variable Analysis）
   - 消除结果未被使用的指令（`STORE_GLOBAL`/`STORE_LOCAL` 除外，因其可能有副作用）
   - 注意处理条件跳转中的死分支

3. **实现拷贝传播（Copy Propagation）**：
   - 将连续的拷贝替换为原始值
   - 处理 `LOAD_LOCAL r1 #0; STORE_LOCAL #1 r1; LOAD_LOCAL r2 #1` → `LOAD_LOCAL r1 #0` 的拷贝链

4. **实现代数化简（Algebraic Simplification）**：
   - `x + 0` → `x`
   - `x - 0` → `x`
   - `x * 0` → `0`
   - `x * 1` → `x`
   - `x / 1` → `x`
   - `x - x` → `0`
   - `x * 2` → `x + x`（强度削弱）
   - `x / 2` → `x >> 1`（强度削弱）
   - 注意处理除数为零的情况

5. **实现优化 Pass 框架的迭代执行**：
   - 多个优化 Pass 需要迭代执行直到收敛
   - 例如：常量折叠 → 死代码消除 → 拷贝传播 → 再次常量折叠
